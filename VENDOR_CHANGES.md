# VENDOR_CHANGES —— 对 verl 的二次开发登记

> 本文件记录我们对上游 verl 做了哪些改动，用于版本升级时做「三方对比」，
> 判断上游的新变化需不需要吸收。**每次改动 verl 相关代码，请更新这里。**

当前跟踪的上游版本见根目录 [`VERL_VERSION`](./VERL_VERSION)。

---

## 一、整体替换的模块（放在 `my_verl/`，`verl/` 保持纯净）

这些模块我们完全用自己的实现，通过导入覆盖顶掉上游版本，升级时 `verl/` 不产生冲突，
但需在升级后确认上游接口是否变化。

| 我们的实现 | 对应上游模块 | 说明 / 为什么替换 | 基于上游版本 |
|-----------|-------------|------------------|-------------|
| `my_verl/workers/xxx.py` | `verl.workers.xxx` | （示例）rollout 逻辑整体重写 | v0.8.0 |

## 二、直接在 `verl/` 里小改的模块（升级时可能冲突）

这些改动直接落在 `verl/` 内部，升级时可能与上游冲突，需手动合并。

| 文件 | 改了什么 | 为什么 | 基于上游版本 |
|------|---------|--------|-------------|
| `verl/verl/trainer/...` | （示例）加了一个 config 字段 | 支持自定义 reward | v0.8.0 |

---

## 升级流程（配合 `sync_verl.sh`）

1. 升级并同步：
   ```bash
   ./sync_verl.sh --set <新版本>     # 例如 v0.9.0
   ```
2. 对「一、整体替换」的模块：检查上游对应模块从旧版到新版的变化，决定是否移植：
   ```bash
   git diff <旧版tag> <新版tag> -- verl/verl/workers/xxx.py
   ```
3. 对「二、小改」的模块：解决 subtree pull 产生的冲突，`git add verl/ && git commit`。
4. 更新本文件里的「基于上游版本」列。
5. 跑测试，确认无回归后 `git push`。
